# ADR-0002：文档信息标准化优先于功能开发

- 状态：Accepted
- 日期：2026-07-17
- 决策范围：ME-System 第一阶段开发主线

## 背景

ME-Who 与 ME-Brain 后续都需要从文档、对话、Git、Zotero、邮件、表格、幻灯片及其他资料中提取信息。若没有稳定的文档信息标准，后续的检索、知识图谱、上下文编译、Hermes 集成和人工治理都会依赖具体解析器产生的临时数据结构。

这会导致：

- 不同来源生成互不兼容的数据；
- 更换解析器时需要重建上层业务；
- 文档结构、原文位置和派生事实之间无法稳定追溯；
- 摘要、向量和图谱结果容易被误认为权威事实；
- ME-Who 与 ME-Brain 分别开发重复的输入处理逻辑；
- Hermes 无法获得稳定、一致、可解释的上下文。

## 决策

ME-System 第一阶段把 **文档信息标准化** 设为最高优先级。

开发顺序调整为：

```text
原始来源与版本
→ 通用文档标准包
→ 结构与资产标准化
→ 来源与质量校验
→ 领域候选映射
→ ME-Brain / ME-Who 权威对象
→ 检索、Context Pack 与 Hermes 集成
```

在文档标准包稳定前，不优先开发复杂知识图谱、完整用户画像、通用 Agent 接入、可视化画布或大规模自动化功能。

## 核心原则

1. **原始资料不可替代**：任何标准化、摘要和抽取均不能覆盖原文件。
2. **先结构、后语义**：先稳定章节、段落、表格、图片、引用和位置，再提取决策、需求、偏好等领域对象。
3. **通用标准与领域模型分离**：文档标准包不直接包含 ME-Brain 或 ME-Who 的业务结论。
4. **解析器可替换**：RAGFlow、Docling、MinerU、自研解析器或其他工具均通过 Adapter 接入。
5. **全部派生可追溯**：领域对象必须能够回溯到文档版本和具体内容节点。
6. **允许部分成功但禁止静默丢失**：无法识别的内容必须进入质量报告和缺口清单。
7. **标准包可重新生成**：派生结构应能够由原始来源确定性或可审计地重建。
8. **机器格式与人类投影并存**：权威交换格式采用结构化数据，同时提供 Markdown 等可读投影。

## 对 Hermes 集成的影响

Hermes 仍是第一阶段主要使用方，但 Hermes 集成暂时只稳定接口边界：

- Hermes 查询 ME-Brain 和 ME-Who 的 Context Pack；
- Hermes 不直接依赖具体文档解析器；
- Hermes 需要下钻证据时，通过稳定的 `document_id`、`version_id` 和 `node_id` 获取原文；
- Hermes 提交的新增信息先作为候选，不直接修改标准文档包或权威领域数据。

## 竞品能力的使用方式

竞品优势将按功能模块逐项吸收：

- RAGFlow、Docling、MinerU：复杂文档解析能力；
- Graphiti：时间与来源关系；
- Codebase-Memory：结构优先和渐进式装载；
- Basic Memory：人类可读投影与 MCP 交互；
- LightRAG：图、全文与向量混合检索；
- ChatGPT Memory、Supermemory、Mem0：用户记忆体验与候选抽取。

任何竞品组件都不得直接定义 ME-System 的权威文档格式和领域模型。

## 后果

### 正面影响

- 两条产品线共享稳定输入基础；
- 上层功能可以并行且可替换；
- 文档证据链、版本和质量可统一管理；
- 更容易进行 Token、速度和准确率评测；
- 后续领域包不需要重复解决文件解析和定位问题。

### 代价

- 第一阶段可见功能较少；
- 需要投入较多时间设计 Schema、定位规则和验证语料；
- 不同格式的无损标准化难度较高；
- 必须建设质量报告和人工复核机制。

这些代价属于基础设施必要投入，不能通过先做演示型 RAG 或知识图谱来规避。
